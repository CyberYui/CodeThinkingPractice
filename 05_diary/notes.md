# 05 日记本 - 笔记

## 我的设计思路
    类似于todolist的设计，创建了日记条目的类，包含实时生成的时间戳和内容属性。日记本则使用了文档目录的形式来展现，每个日记条目都保存到一个文件中，文件名包含日期和时间以确保唯一性。用户可以通过输入内容来创建新的日记条目，并且可以查看所有的日记条目列表等操作。

## 卡住的地方
    纠结了一下用户在使用查看历史日记的时候，是否要求用户输入一个编号来查看对应的日记内容，还是直接让用户输入一个日期来查看对应的日记内容。最后我选择了前者，因为这样可以更直观地展示所有的日记条目，并且用户可以通过编号快速访问特定的日记内容，只不过对于通过日期查看日记的函数会判断用户是否是从历史记录里访问来的，如果是则直接获取对应文件的路径并查看指定日期的日记，否则则让用户输入一个日期，获取对应文件的路径并查看指定日期的日记。

## 学到的
    view_specific_date 用 int choice 做参数判断来源，但伪代码里 if choice: 判断的是"有没有传参"，实际应该用 None 做默认值更清晰一些。
    view_history 里用户输入编号后直接 int(input()) 没做异常处理，老问题
    文件名用完整日期时间会冲突（同一秒写两篇会覆盖），用日期就够了，但如果用户想写多篇同一天的日记就会有问题，可以考虑在文件名里加上一个随机数或者时间戳来确保唯一性。

## 和AI代码的差异
    AI代码通过引入datetime模块中的strftime函数来生成包含日期和时间的文件名，确保每篇日记都有一个唯一的标识符。我的代码则计划使用python自带的datetime模块来获取当前日期和时间，并将其格式化为字符串，以便在创建日记条目时使用。
    AI代码在保存文件的时候做了文件存在的检查，并且沿用了之前练习中使用with open() as f的方式来处理文件操作，这样可以确保文件在使用完毕后正确关闭，避免资源泄漏。
    AI代码在查看历史日记的时候做了两层存在判断，分别是目录是否存在和文件是否存在，这样可以避免程序崩溃。
    AI代码在view_specific_date函数中传入了choice=None作为默认参数，很清晰地判断了用户是否是从历史记录里访问来的，这样可以更好地控制函数的行为。另外限定了文件夹名为diary，这样可以更好地组织文件结构。且对于用户输入的日期也做了格式检查，确保用户输入的日期格式正确。

## 运行结果
    运行结果符合预期，用户可以成功创建新的日记条目，并且可以查看所有的日记条目列表等操作。

## 下次改进
    针对此类较为繁琐的小程序，已经可以开始考虑要不要添加UI界面了，毕竟命令行界面对于用户来说可能不够友好，尤其是对于不熟悉命令行操作的用户来说。可以考虑使用一些Python的GUI库，比如Tkinter或者PyQt来创建一个简单的图形界面，让用户可以更直观地操作日记本。
    另外在文件命名方面，可以考虑在文件名中加入一个随机数或者时间戳来确保唯一性，避免同一秒写两篇日记时发生覆盖的情况。
    关于文件目录的组织也可以进行用户配置，让用户可以选择自己喜欢的目录结构来保存日记条目，这样可以更好地满足不同用户的需求。
